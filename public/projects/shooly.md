## 問題意識：生活服務市場的結構性痛點

Shooly 的誕生，源自一個非常具體且普遍的生活問題。

對於在外地租屋的學生與上班族而言，當 **水電、電腦、紗窗等設備臨時損壞** 時，往往面臨：

- 不熟悉當地可的水電技師  
- 價格資訊不透明、服務品質難以評估  
- 媒合流程冗長，溝通成本高  

另一方面，對於水電技師而言，傳統產業結構相對封閉：

- 曝光管道有限，難以累積可被搜尋的信譽  
- 缺乏數位化工具管理客源與評價  
- 無法有效觸及年輕世代使用者  

**Shooly 的核心問題意識在於：  
如何透過科技，讓民眾「快速找到對的人」，也讓技師「被看見、被信任」。**

## 核心理念：以對話式介面重構服務媒合流程

Shooly 採用 **LINE Chatbot 作為主要互動入口**，以使用者最熟悉的即時通訊平台，重新設計生活服務媒合體驗。

{%youtube https://www.youtube.com/watch?v=Ynk_URA8BBw %}

設計原則包含三個關鍵方向：

1. **零學習成本**：不需下載新 App，直接透過 LINE 使用  
2. **流程極簡化**：以點選與對話引導取代複雜表單  
3. **資訊透明化**：技師評價、價格、服務範圍清楚可見

## 系統架構與技術實作

### 整體架構

Shooly 為一個 **以 Chatbot 為前端、API 為核心的分散式系統**：

- **前端**
  - LINE Chatbot
  - LINE LIFF（提供較高權限的互動與定位功能）

- **後端**
  - Ruby on Rails Server
  - RESTful Web API

- **雲端架構**
  - Chatbot Server：Heroku（高頻推播、快速部署）
  - Web API / 資料服務：GCP（安全性與擴充性考量）

此架構在成本、效能與安全性之間取得實務上的平衡，並具備未來完整遷移至 GCP 的彈性。

## 核心功能設計

### 1. 快速媒合

使用者可透過 **項目 × 地區** 的點選流程：

- 即時搜尋附近可到府服務的技師  
- 檢視技師的服務項目、地區、資歷、證照與評價  
- 快速完成預約與聯繫  

---

### 2. 一對一即時諮詢

Shooly 設計了 **不需加好友的技師對話機制**：

- 由 Chatbot 作為溝通橋梁  
- 伺服器端先配對雙方 User ID  
- 透過 LINE Push API 實現雙向即時對話  

使用者可直接傳送文字、照片或影片，讓技師進行遠端判斷與報價，大幅降低溝通成本。

---

### 3. 即時情境資訊整合

平台整合多項外部 API，提升服務決策品質：

- **Google Maps API**
  - 技師位置、路線與預估抵達時間  

- **中央氣象局 API**
  - 提供預約時段的天氣資訊  
  - 協助判斷是否適合進行戶外或清洗工程

## 實務挑戰與工程解法

### 使用者定位限制

由於 LINE 平台限制，Chatbot 無法在背景即時取得使用者位置。  
Shooly 的解法為：

- 透過 **LIFF 機制**  
- 在使用者每次開啟介面時紀錄位置  

並規劃未來以原生 App 方式進一步解決此限制。

---

### 推播成本問題

一對一對話仰賴 LINE Push API，推播成本較高。  
團隊在系統設計上透過：

- 對話流程最小化  
- 僅在必要節點推播  

來降低實際營運成本。
